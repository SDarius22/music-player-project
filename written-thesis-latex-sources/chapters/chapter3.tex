\section{Architecture}
\label{sec:architecture}

\indent \par This part starts with the static shape of the system, before any request is sent and before any peer connection is opened. Its role is to turn the abstract roles from Chapter \ref{chap:ch2} into concrete components that can later be followed in the implementation flow. It first states the requirements that the prototype has to satisfy, because those requirements explain why the code is split the way it is. It then fixes the development and deployment context, since the same design has to run on a local machine during testing and on several backend replicas in production. From there, the section moves into the database schema, the backend service layers and the client architecture. The diagrams are not meant as decoration, they are the map that makes the later runtime story easier to follow.

\subsection{System Requirements}
\label{sec:requirements}

\indent \par The requirements were not written down at the beginning and then frozen. They evolved as the implementation forced decisions, and as some assumptions from Chapter \ref{intro} turned out to be either too optimistic or too pessimistic in practice. The paragraphs below capture the requirements at the moment of writing, separated into functional and non-functional ones. The split is the conventional one: the functional set says what the application must do, while the non-functional set says how well it must do it.

\indent \par The functional requirements set the user-facing baseline. A user must be able to register and log in with an email through a verification code, passwordless, with short-lived access tokens and a long-lived refresh token handling the re-authentication afterwards. The catalogue must be browsable, searchable and sortable across artists, albums and songs, with streamable content shared between all users and uploaded content private to the uploader. Playback must cover the full set of controls (play, pause, seek, next, previous), together with shuffle, repeat and an editable queue, while library management must let the user like a song, with the changes propagating across devices. Playlists must be creatable, editable, and deletable, except for the built-in playlists (such as the queue, most played, or recently played) which are marked indestructible. Audio files must be uploadable from disk, with backend chunk-level deduplication so repeated 64 KiB blocks are stored only once. The backend should also expose a negotiated chunk upload path for clients that compute manifests locally. Peer-assisted streaming must use other currently-listening clients when possible, falling back to the server when no peers are available, and the playback position together with the shuffle, repeat, auto-play and auto-play recommendation page values must be restored when the user opens the application on another device.


\indent \par The non-functional requirements describe how well the application has to do what the previous paragraph says. The system has to reduce the per-listener egress compared to a pure client-server delivery, in line with the cost model from Chapter \ref{chap:ch2}, and a single source tree must run on Linux, macOS, Windows, Android, iOS and the web browser. The backend must support several replicas sharing a database and a Redis instance, with the cross-replica routing transparent to the client. Peers are not trusted, so every chunk is verified before reaching the audio decoder, and the server fallback is always available so that the user never sees an error when peers are absent or slow. Finally, both the backend and the frontend must be testable in isolation, with as few moving parts as possible during the test runs.


\subsection{Specification}
\label{sec:specification}

\indent \par The prototype was developed on an Apple MacBook Pro with an M2 Pro chip, 16 GB RAM and macOS. The toolchain is Java 25 with Maven and Spring Boot 4.0.3 on the backend, Liquibase for schema migrations, Flutter on the client with the \texttt{provider} package for dependency injection, ObjectBox on desktop and mobile, in-memory catalogue and chunk repositories on the web, and browser local storage for web settings. Docker and Docker Compose host the local PostgreSQL 18 and Redis Alpine instances. The production deployment uses one home-server with an Intel i5-6500, 8 GB RAM and Ubuntu Server 24.04, runs two Spring Boot replicas behind a reverse proxy and shares the same PostgreSQL instance, Redis instance and music data files. This server is also used for hosting the frontend on the web and for hosting the mail SMTP server necessary to authenticate users, as defined in the previous section.


\subsection{Database Schema}
\label{sec:db-schema}

\indent \par The persistence layer is built on PostgreSQL, inside a dedicated \texttt{music\_library} schema, and is managed through eight Liquibase changesets at the moment of writing. JPA is configured in \texttt{validate}-only mode, so any structural change must come through a new changeset. This keeps the schema under version control and makes every migration explicit and ordered, rather than inferred from the entity classes at start-up. The diagrams that follow group the tables by concern: the media catalogue and chunk store first, then the per-user state, and finally the auxiliary tables.

\indent \par Figure \ref{fig:db-schema-media} shows the tables that hold the media catalogue and the chunked audio data. Artists and albums share a many-to-many relation through the \texttt{album\_artists} join table, while \texttt{songs} keeps a legacy single \texttt{artist\_id} column for tracks with one author. Every song has a unique \texttt{file\_hash} (SHA-256 of the original file) that is the stable identifier from upload to playback. Because that identifier is derived from the file content, the same upload always resolves to the same row, which is what makes the playlist references and the chunk mapping stable across re-imports.

\begin{figure}[htbp]
\centering
\includegraphics[width=\textwidth]{figures/db_schema_media.png}
\caption{Tables for the media catalogue and the chunked audio storage.}
\label{fig:db-schema-media}
\end{figure}

\par The chunk store is split in two tables. \texttt{chunks} holds one row per distinct 64 KiB block, keyed by \texttt{content\_hash}, with its size and filesystem path. \texttt{song\_chunks} maps a song to its ordered list of chunks. This separation is what enables deduplication: two songs that share a 64 KiB block reference the same row in \texttt{chunks}, and the physical file lives on disk only once. The chunks themselves are kept on the filesystem rather than as PostgreSQL blobs, because large binary columns cause memory pressure on both reads and backups.


\indent \par The user-side tables, drawn in Figure \ref{fig:db-schema-user}, keep the per-user state. \texttt{users} holds the email, the role (\texttt{USER} or \texttt{ADMIN}) and the provider (\texttt{LOCAL} or \texttt{GOOGLE}, the latter is reserved for a future OAuth integration). \texttt{user\_library} records the user's relation to a song (liked flag, play count, total play duration, timestamps). Keeping this relation in its own table rather than as columns on the song lets the same catalogue entry carry different state for every user without duplicating the song itself.

\begin{figure}[htbp]
\centering
\includegraphics[width=\textwidth]{figures/db_schema_user.png}
\caption{Tables for users, library, playback state, playlists and the playlist-to-song mapping.}
\label{fig:db-schema-user}
\end{figure}

\par The \texttt{user\_playback\_state} is kept intentionally minimal. After several iterations, the queue and the current song were removed from the server-side state because keeping a shared queue across devices proved more trouble than it was worth. What remains is the position in seconds, shuffle and repeat flags, the auto-play preference and the position in the auto-play recommendation feed. The queue is reconstructed locally from the 'Queue' playlist and the user's last played song, stored in the 'Recently Played' playlist.

\par Playlists live in two tables. \texttt{playlists} keeps the metadata and the \texttt{indestructible} flag. \texttt{playlist\_songs} references songs by \texttt{song\_file\_hash} rather than by id, which makes the playlist content portable across re-imports of the same audio file. The \texttt{indestructible} flag marks the built-in playlists, such as the queue and the recently-played list, so the interface can refuse to delete them while treating everything else as ordinary user data.


\indent \par Two auxiliary tables complete the schema (Figure \ref{fig:db-schema-auxiliary}). \texttt{verification\_codes} holds the short-lived email codes consumed by the authentication endpoints, and \texttt{chunk\_stats} holds the per-session delivery telemetry used by the evaluation in Chapter \ref{chap:ch4}. Neither table is part of the catalogue or the user library, which is why they are kept separate from the two groups above. The \texttt{chunk\_stats} rows in particular are write-only from the client's point of view: the application records where each chunk came from, and only the evaluation ever reads them back.

\subsection{Backend Service Architecture}
\label{sec:backend-arch}

\indent \par Figure \ref{fig:backend-architecture} shows the architecture of a single Spring Boot replica together with its external dependencies. The application is API-first: the source of truth for the REST surface is the \texttt{api.yml} OpenAPI document, from which the controllers and the DTO classes are generated at compile time. The business logic lives in the service layer. The controllers stay thin, delegating to the services immediately, so that the generated layer never carries behaviour that a regeneration would overwrite.

\begin{figure}[htbp]
\centering
\includegraphics[width=\textwidth]{figures/backend_architecture.png}
\caption{Backend service architecture and its external dependencies.}
\label{fig:backend-architecture}
\end{figure}

\indent \par The security stack is built around Spring Security and a custom JWT filter that extracts the bearer token, validates it against the JWT service, and populates the \texttt{SecurityContext}. The authentication itself is passwordless and code-based: the client calls \texttt{POST /auth/send-code} with an email, the backend stores a 6-digit code in \texttt{verification\_codes} and dispatches it through SMTP, then \texttt{POST /auth/verify} consumes the code and returns an access/refresh pair. Access tokens are short-lived (minutes), refresh tokens are long-lived (days), and the client refreshes well before expiry through a periodic timer. The passwordless flow keeps no password hashes on the server, so the worst a database leak can expose is a set of already-expired one-time codes.

\indent \par The catalogue services (\texttt{AlbumService}, \texttt{ArtistService}, \texttt{SongService}, \texttt{PlaylistService}) expose CRUD operations on top of JPA repositories, plus search and ordering. They also enforce a small set of invariants that the schema cannot enforce on its own: the indestructible flag, the per-user uniqueness of playlist names, and the visibility rules for user-uploaded content on the song-listing, playlist and streaming paths. Album and artist list endpoints remain catalogue-level views, their detail endpoints filter the songs they expose, but the list entries themselves are not hidden solely because they currently contain only another user's private upload. Search is implemented with SQL \texttt{LIKE} on names, joined where needed to keep unrelated song hits out. The smaller \texttt{RecommendationService} returns a page of recommended songs based on listening history. The implementation is intentionally simple, but the interface allows a smarter ranking to be plugged in later.

\indent \par The current administrative upload screen uses a full-file REST upload. The client computes the SHA-256 of each file and sends it together with the bytes and the visible metadata to \texttt{POST /songs}. The backend cuts the bytes into 64 KiB chunks, stores only physical chunks whose content hash is new, and creates the \texttt{song\_chunks} mapping. This path does not save upload bandwidth for duplicate chunks, but it does keep the server-side chunk store deduplicated.

\par The backend also exposes a negotiated upload protocol for clients that compute the manifest locally. \texttt{POST /songs/negotiate} sends the file metadata and the manifest of chunk hashes, the backend replies with the indices it does not have yet. \texttt{POST /songs/\{fileHash\}/chunks/\{chunkIndex\}} then transfers only the missing chunks. Once all of them arrive, the same song creation path runs. This second path is useful for resumable or bandwidth-aware clients, but it is reserved for future work and not the path taken by the current graphical upload screen.

\indent \par The streaming service exposes two endpoints. \texttt{GET /stream/\{fileHash\}/manifest} returns the chunk count, the chunk size (64 KiB), the total byte length and the ordered list of SHA-256 hashes. \texttt{GET /stream/\{fileHash\}/chunk/\{chunkIndex\}} returns the raw bytes of a single chunk. Access control is enforced at the query level: a wrong user gets a 404 rather than a 403, which avoids leaking the existence of someone else's private uploads.

\indent \par \texttt{PlaybackStateService} reads and writes the \texttt{user\_playback\_state} row through \texttt{GET /playback} and \texttt{PUT /playback}. The DTO carries five values: \texttt{positionSeconds}, \texttt{shuffle}, \texttt{repeat}, \texttt{autoPlay} and \texttt{autoPlayRecommendationsPage}. Other devices will pick up the new state by re-fetching it on the next start. A cross-replica push was considered but left out because pull-on-demand was enough for the multi-device experience. Library mutations (likes, play count, last played) take a different path: each one is a \texttt{PATCH /songs/\{fileHash\}} call with only the changed fields.

\indent \par The signaling component extends Spring's \texttt{TextWebSocketHandler} and processes the messages sent to \texttt{/ws/signaling}. It maintains three in-memory concurrent maps (a session registry, a user index that records which sessions belong to an account, and a peer index for direct routing), and uses \texttt{PeerTrackingService} on top of Redis to track which chunks each peer currently has cached. The Redis keys are indexed both by song and by peer id, so discovery touches the peer map for only the requested song and disconnect cleanup touches only the songs registered by the departing peer. After the initial \texttt{AUTH} message, six client-initiated message types are handled: \texttt{REGISTER\_CACHE} merges newly cached chunks into the peer's existing entry, \texttt{DISCOVER\_PEERS} returns a \texttt{PEER\_BUFFER\_MAP} of all other peers with chunks for the requested song, \texttt{OFFER}, \texttt{ANSWER} and \texttt{ICE\_CANDIDATE} are forwarded to their \texttt{targetId} either directly (target on this replica) or through Redis (target on another), and \texttt{PING} is a keepalive that prevents intermediate boxes from dropping the WebSocket as idle.


\subsection{Client Architecture}
\label{sec:client-arch}

\indent \par State management and dependency injection on the client rely on the \texttt{provider} package. The major services (audio, WebRTC, REST clients and the chunk service factory) are registered at the root of the widget tree inside \texttt{AbstractApp} and retrieved by descendants through \texttt{context.read<T>()}. The chunk service is the exception: it is instantiated per song because its state (manifest, RAM cache, deduplication map and peer completers) belongs to one audio file. The audio service therefore receives a factory function rather than a compile-time dependency. The remaining client decisions worth discussing are the platform-specific repository split and the audio library choice.

\indent \par Desktop and mobile expose a persistent filesystem and support native libraries, while the browser does not expose the same local storage model. The application defines abstract repository interfaces (songs, chunk cache, settings) with platform-specific implementations. The ObjectBox-backed repositories are used on Linux, macOS, Windows, Android and iOS. ObjectBox was chosen over SQLite because it requires no application-level migrations and its Dart code-generation tooling fits cleanly into Flutter, plus the performance boost is not something that we wanted to oversee. The native chunk cache itself sits on disk as raw binary files rather than inside ObjectBox, again because of memory pressure during reads. The web platform uses in-memory \texttt{Map} and \texttt{List} structures for the catalogue and chunk cache, accepting that locally cached chunks are lost when the tab closes and the browser is therefore a weaker seeder than a desktop client. Web settings are the exception: they are persisted through browser local storage so user preferences survive a page reload.

\indent \par The natural Flutter choice is \texttt{just\_audio}, which exposes a \texttt{StreamAudioSource} abstract class that lets the application provide bytes on demand by range. On Linux and Windows, however, the underlying decoders (GStreamer and Windows Media Foundation) expect a seekable sequential source and stall when seeking through a dynamic stream. The workaround was to switch to \texttt{media\_kit} on Linux, macOS and Windows, as it wraps \texttt{libmpv} with an FFmpeg demuxer that is comfortable with random access. The client now uses \texttt{just\_audio} on Android, iOS and the web, and \texttt{media\_kit} on the three desktops platforms. Both sit behind the same interface so the UI and chunk pipeline do not have to know which is in use. However, on the web, the browser audio element issues HTTP range requests, which leaves no Dart-level hook to serve them the solution is a Service Worker that intercepts requests under a known URL scheme and assembles the response from the chunk cache.


\section{Implementation}
\label{sec:implementation}

\indent \par With the structure established, this part follows the code while it is moving. The order is intentionally close to the order in which a user experiences the application: the app opens, authentication and settings are restored, a song is selected, the manifest is loaded, peers are discovered, chunks are fetched, verified and cached, and the audio player receives bytes without knowing whether they came from the server or from another device. After the main playback path is clear, the section looks at the surrounding machinery that keeps the system usable in practice: WebRTC peer management, peer ranking, upload limits, cross-replica signaling, shutdown behaviour and testing. The limitations are kept in this same part because they belong to the implementation as it really exists, not to an ideal version of it.

\subsection{Application Bootstrap}
\label{sec:bootstrap}

\indent \par The bootstrap covers what happens between the moment the application opens and the moment the main scaffold is visible (Figure \ref{fig:flow-startup-restore}). \texttt{main.dart} initialises the local database, the audio plugin and the platform-specific bits, and builds the \texttt{MultiProvider} tree from \texttt{abstract\_app.dart}. The \texttt{WebRTCService} is the only eagerly created provider (\texttt{lazy: false}), but it opens the signaling WebSocket only after \texttt{AuthService} has an access token. An auto-login step follows, the loading screen calls \texttt{tryAutoLogin()} on the \texttt{UserProvider}, which reads a refresh token from secure storage and exchanges it through \texttt{POST /auth/refresh} for a new access token, with a periodic timer in \texttt{AuthService} refreshing every ten minutes afterwards. When that token is saved, the auth listener inside \texttt{WebRTCService} connects to \texttt{/ws/signaling} and authenticates the socket. Finally, the settings service calls \texttt{GET /playback} and applies the five values from the DTO to the local audio settings before the application navigates to the main scaffold. The library content (recently played, favourites, playlists) is loaded lazily as each list is opened.

\subsection{Streaming Pipeline}
\label{sec:streaming-pipeline}

\indent \par This section follows what happens when the user presses play. \texttt{AppAudioService.play()} invokes \texttt{\_loadIndex(int index)} for the current queue position and calls the source factory: on native targets the factory builds a \texttt{P2PChunkedAudioSource}, while on the web the source is a base-relative URL of the form \texttt{p2p-stream/\{fileHash\}} intercepted by the Service Worker. From there the flow continues with the manifest fetch and peer discovery, the chunk loop itself, and the caching, fan-out and prefetch machinery that the loop relies on. The same source abstraction sits in front of both platforms, so the rest of the pipeline does not need to know whether it is feeding a native decoder or the Service Worker.

\indent \par On the first byte-range request, the audio source asks the chunk service for the chunk that contains those bytes. The service triggers \texttt{loadManifest()}, which calls \texttt{GET /stream/\{fileHash\}/manifest} and receives the chunk count, the chunk size and the SHA-256 hashes. Once the manifest is available, it sends a \texttt{DISCOVER\_PEERS} message asynchronously. The backend replies with the buffer map of every known peer for the file, including the exact chunk indices advertised by each peer. The service also reads the local cache and sends \texttt{REGISTER\_CACHE} with any indices it already holds. If the buffer map is non-empty, the WebRTC service stores the peer-to-chunk map and starts the offer/answer/ICE handshake while the chunk loop continues to satisfy playback from the cache, server prefix or fallback path (Figure \ref{fig:flow-press-play-discovery}).

\begin{figure}[htbp]
\centering
\includegraphics[width=\textwidth]{figures/flow_press_play_discovery.png}
\caption{Manifest fetch, peer discovery and the start of the WebRTC handshake.}
\label{fig:flow-press-play-discovery}
\end{figure}

\indent \par For each chunk the decoder needs, the chunk service runs the decision tree in Figure \ref{fig:flow-press-play-chunk}. The local cache is consulted first. On a miss, the implementation defines

\begin{equation}
  N_{\text{prefix}} = \max\!\bigl(1,\ \text{round}(N_{\text{total}} \times 0.05)\bigr)
  \label{eq:prefix-impl}
\end{equation}

as the number of prefix chunks (5\% of the song, with a minimum of one). Foreground playback requests below this threshold are forced to the server, guaranteeing that playback can start before any peer connection is open. Above the threshold, only peers that advertised the requested chunk are tried first through the staggered fan-out described below, with a 1700 ms playback timeout. The separate prefetch path uses a 2500 ms peer timeout and may try peers for future prefix chunks too, because those requests are not on the immediate playback path. Every received chunk is verified by recomputing its SHA-256 against the manifest hash, with a mismatch causing a server retry when the bad bytes came from a peer, and a second mismatch is treated as a hard integrity error. After a successful fetch, the chunk is written to RAM and disk caches, a \texttt{REGISTER\_CACHE} update goes back to the signaling server, and the delivery source is recorded for the periodic statistics reporter.

\begin{figure}[htbp]
\centering
\includegraphics[width=\textwidth]{figures/flow_press_play_chunk.png}
\caption{Chunk streaming loop.}
\label{fig:flow-press-play-chunk}
\end{figure}

\par The \texttt{P2PChunkedAudioSource} that sits on top of this loop translates each audio-player range $[s, e)$ into the chunk indices $i_{\min} = \lfloor s / C \rfloor$ and $i_{\max} = \lfloor (e - 1) / C \rfloor$, iterates between them and yields the appropriate slice of each chunk to the output stream. The response declares range-request support and a precise \texttt{Content-Length}, without which the audio plugin will skip to the next song or stop working altogether. A single range request can therefore span several chunks, each of which is resolved through the same cache, peer and fallback path before its bytes are spliced into the response. The source is the only place that knows about byte offsets, everything beneath it works purely in whole chunks.

\indent \par Three caching layers are stacked in order of access speed. The hot RAM cache holds at most 15 entries keyed by chunk index, with FIFO eviction, absorbing the repeated reads from overlapping range requests, especially during seeks. The disk cache stores chunks as binary files in a per-song directory. Writes are asynchronous, so the caller receives the data immediately while the disk write runs in the background. The third layer is an active-request deduplication map from chunk index to \texttt{Future<Uint8List>}: two concurrent callers requesting the same chunk attach to the same future, which prevents duplicate network or peer requests during seeking. Incoming binary chunk responses are routed through \texttt{ActiveChunkRouter}, which maps a file hash to the currently active chunk manager. The router replaces an older manager for the same file hash but does not evict managers by capacity, so it does not drop a response merely because the small factory cache has rotated.

\par When peers are available for the requested chunk, the chunk service asks the WebRTC service for the peers that advertised that chunk, sorted by score (Section \ref{sec:webrtc-client}), and keeps at most the top three. The request is issued to the best peer first. If the shared completer has not completed after 150 ms, the second peer is tried, and after another 150 ms the third peer is tried. If a peer's data channel is not open yet, the WebRTC service queues the request until the channel opens. The first response completes the chunk request, while later duplicate responses no longer have an active completer and are ignored. This staggered fan-out keeps the common-case latency low while still triggering fast recovery when the best peer is slow.

\par To prevent the decoder from starving, the chunk service maintains a constant 16-chunk lookahead. After each \texttt{getChunk(i)} call, fetches for $i+1$ through $i+16$ are initiated asynchronously, gated by the deduplication map. Sixteen chunks is about 1 MiB or 25 seconds at 320 kbps, large enough that even a worst case of one peer timeout per chunk for all 16 prefetches would not exhaust the buffer before the server fallback returns. The window is a fixed size rather than a growing read-ahead, so the memory it costs is bounded regardless of how long the track is.

\subsection{WebRTC Peer Management}
\label{sec:webrtc-client}

\indent \par The WebRTC service manages the lifecycle of every peer-to-peer connection. It is constructed with a reference to the signaling WebSocket and two callbacks: one for when a binary chunk arrives over a data channel, and one for when a peer requests a chunk this device may be able to serve. The two callbacks are the only coupling between the transport and the chunk pipeline, which keeps the WebRTC code unaware of how chunks are cached or verified. Everything else it does, the offers, the answers and the ICE exchange, is in service of opening and maintaining those data channels.

\indent \par Peer connections use the standard offer/answer handshake. When the buffer map arrives, the service creates a new initiator-mode peer connection for each newly discovered peer, configured with two STUN servers (\texttt{stun.l.google.com:19302} and \texttt{stun1.l.google.com:19302}) so that ICE still completes if one is unreachable. An ordered reliable data channel labelled \texttt{chunk\_transfer} is opened on top, mirroring TCP semantics, which is necessary because a partial 64 KiB block cannot be verified against the manifest hash. The offer carries an \texttt{offerId} that the remote echoes back in its answer, which is used to discard stale answers, and glare (both sides offering simultaneously) is resolved by keeping the offer of the peer with the lexicographically larger device id.

\par Once the data channel is open, requests are sent as UTF-8 text messages of the form \texttt{REQUEST\_CHUNK:\{fileHash\}:\{chunkIndex\}}, while responses are binary. If the chunk fits within the safe DataChannel limit of about 16 KiB, a single packet is sent with a 68-byte header (the 64-character file hash followed by a 4-byte big-endian chunk index) and the audio payload after it. Otherwise the chunk is fragmented, with each fragment carrying a 77-byte header starting with the magic sequence \texttt{P2PF}, so the receiver can distinguish fragments from single-packet messages, and the pieces are reassembled per \texttt{(peerId, fileHash, chunkIndex)} tuple. Fragmentation was added because some WebRTC implementations, notably on Android, silently refuse messages larger than 16 KiB. Further limitations are described in Section \ref{sec:limitations}.

\indent \par Each remote peer is tracked with exponentially weighted moving averages for round-trip time and throughput, plus cumulative success and failure counters. The round-trip time is refreshed every 30 seconds through a ping/pong message over the data channel, while throughput is updated on every successful chunk response as the number of received bytes divided by the request-to-response time. The score is

\begin{equation}
  \text{score} = \frac{\text{successRate} \times \text{throughput} \times 1000}{\text{RTT} + 1},
  \label{eq:peer-score}
\end{equation}

where \texttt{successRate} is the ratio of successful chunk responses to all recorded attempts, defaulting to 0.5 before any result is known. The $+1$ in the denominator prevents division by zero, and the smoothing factor $\alpha = 0.2$ gives recent RTT and throughput measurements 20\% weight and the previous estimate 80\%. The chunk service first filters peers to those that advertised the requested chunk and then uses at most the top three peers from this ranking in its staggered fan-out.

\indent \par On mobile, acting as a seeder consumes the user's data allowance. The WebRTC service checks two settings before sending a chunk: the peer network mode (Wi-Fi only, cellular only, or both), checked through the connectivity plugin, and a monthly upload cap configurable separately for Wi-Fi and cellular and stored with a \texttt{year * 100 + month} integer so it survives restarts. A mismatch on either silently skips the send and the requester falls back to the server. On desktop and on the web these checks are bypassed.


\subsection{Audio Service and Playback State}
\label{sec:audio-service}

\indent \par The \texttt{setQueueAndPlay} method atomically replaces the queue and calls the source factory, which branches on three cases: a song with a local file path becomes a \texttt{Uri.file} source with no network I/O (this was intentionally left as a future work), on the web the source is the Service Worker URL, otherwise a \texttt{P2PChunkedAudioSource} triggers the manifest fetch on the first range request. Both the normal and the shuffled queues are kept in sync, and when shuffle is toggled the shuffled queue is rebuilt with the current song pinned to the front. A small watchdog runs in parallel and reloads the audio source if the playback position fails to advance for a second while the player is nominally playing, which forces fresh range requests and acts as a platform-independent safety net against the occasional silent stall. Pinning the current song to the front of the rebuilt queue means a shuffle toggle never interrupts what is already playing, only what comes after it.

\par Every 30 seconds, while playback is active, the audio service serialises position, shuffle, repeat, auto-play and auto-play page to the playback REST endpoint. The same 30-second timer also runs a proactive prefix cache routine that looks at the next five songs in the queue and the previous two (in case the user goes back). The target fraction is larger for nearby songs: the next five queue entries have caps of 25\%, 20\%, 15\%, 10\% and 5\% of their chunks, while the previous two have caps of 10\% and 5\%. The actual amount requested is each cap multiplied by the playback progress through the current song, so little is fetched near the start and more is fetched as the transition approaches. Each target chunk is handed to the normal \texttt{prefetchChunk} path, so it may come from a peer that advertised the chunk or from the server fallback. The result is that nearby queue entries often have warm cache hits before the user gets there, while distant entries do not consume bandwidth too early.


\subsection{Multi-Instance Coordination}
\label{sec:multi-instance}

\indent \par The backend is designed to run as several replicas behind a load balancer. Two different devices may have connected to different replicas, device A to replica 1, device B to replica 2. The coordination uses Redis pub/sub on the dedicated \texttt{signaling:webrtc} channel. WebRTC offer/answer/ICE messages carry a target id: the handler first checks the peer index for a local session, and if the target is on this replica the message is forwarded immediately, otherwise it is published on the signaling channel and every other replica's listener picks it up, looks up its own peer index, and forwards the message if the target is found there (Figure \ref{fig:flow-redis-webrtc}). The mechanism is transparent to the client.

\begin{figure}[htbp]
\centering
\includegraphics[width=\textwidth]{figures/flow_redis_pubsub_webrtc.png}
\caption{Cross-replica routing of WebRTC offer/answer messages through the signaling channel.}
\label{fig:flow-redis-webrtc}
\end{figure}

\par On a graceful WebSocket close, Spring's lifecycle calls \texttt{afterConnectionClosed}. The handler removes the session from the three indices and asks \texttt{PeerTrackingService} to remove the peer's chunk registrations from Redis, so that no other replica will route P2P traffic to its former clients. A hard process crash is less tidy: the handler does not get a chance to run the same cleanup, so other peers may temporarily discover stale entries and then fall back to the server when requests time out. Automatic client reconnection after such a failure is not implemented in the current frontend, it is treated as a limitation below.


\subsection{Application Shutdown}
\label{sec:shutdown}

\indent \par Shutdown is partly explicit and partly delegated to the process and socket lifecycle. This was an intentional scope choice for the prototype: the system being evaluated is the active playback and peer-delivery path, not long-lived background seeding after the application UI has been torn down. The \texttt{AudioProvider} disposes its listeners and the audio player, while \texttt{AppAudioService} and \texttt{WebRTCService} still expose disposal methods for cancelling timers, closing peer connections and closing the signaling socket. These service-level methods are not all wired directly into the Flutter provider tree because the normal WebSocket close already triggers backend cleanup through \texttt{afterConnectionClosed}, and any P2P request that races with shutdown is handled by the same server fallback used for slow or unavailable peers. In other words, the prototype treats shutdown as fail-safe rather than fully deterministic. A production version should add provider-level dispose hooks for the long-lived services, but leaving that wiring out kept the implementation focused on the streaming behaviour measured in Chapter \ref{chap:ch4}.

\subsection{Testing}
\label{sec:testing}

\indent \par The backend is covered by around forty-five JUnit test classes spread across the layers. Repository and controller integration tests run against real PostgreSQL and Redis instances brought up through Testcontainers, so the JPA mappings, the Liquibase schema and the queries are exercised against the same database engine used in production rather than an in-memory substitute. The service, mapper and JWT-filter layers are tested in isolation with mocked collaborators, and the signaling path has dedicated tests for the handler, the Redis cross-replica listener and the peer-tracking service. On the client, around thirty-five \texttt{flutter\_test} files cover the services (chunk fetching, song and playback logic), the providers, the DTO serialisation and the in-memory repositories, with \texttt{mockito} standing in for the REST clients, plus a widget test and an end-to-end song-and-playlist flow test. The peer-assisted delivery path as a whole is exercised by the headless evaluation client of Chapter \ref{chap:ch4}, which drives the real chunk pipeline without the user interface.

\par Two of the more instructive defects surfaced only under this kind of testing, both concurrency faults that a single-client run never triggers. The first was a teardown race: when the application closed, a chunk that finished caching during shutdown called back into the WebRTC service to register itself for sharing after the signaling socket had already been closed, throwing a `cannot add event after closing` error. The fix routes every signaling send through a guard that drops the message once the session is disposed. The second, described in detail in Chapter \ref{chap:ch4}, was found by the swarm-scaling experiment: when several listeners connected to one seeder at the same instant, the backend wrote to a single WebSocket session from several threads at once, which Tomcat forbids. Offers and answers were silently dropped, and most listeners fell back to the server. Wrapping each session in a \texttt{ConcurrentWebSocketSessionDecorator} serialises those writes and removes the bottleneck. Both faults appeared only once the system was driven concurrently, which is the main argument for exercising it that way.

\subsection{Limitations}
\label{sec:limitations}

\indent \par The current prototype has several limitations that are worth mentioning. Some are inherent to the platforms targeted, some are due to choices made to keep the scope of the thesis manageable. None of them invalidates the architecture, but each one is something a follow-up project would have to address before considering the system production-ready. They are gathered here rather than scattered through the chapter so that the gap between the prototype and a deployable product is visible in one place.

\indent \par As described in Section \ref{sec:client-arch}, the \texttt{just\_audio} package does not handle the custom \texttt{StreamAudioSource} interface well on Linux and Windows, because the underlying decoders expect a seekable, sequential data source rather than the on-demand range-based chunk stream that the application provides. The workaround was to switch to \texttt{media\_kit} on Linux, macOS and Windows. The client therefore uses two audio libraries depending on the platform, which has a maintenance cost: every change in the audio service has to be tested on both backends, and any new feature has to be expressible in terms that both can support. Consolidating on a single library that behaves consistently across all platforms would remove this cost, but no such library was available during development.

\indent \par The browser does not expose a hook for the Dart code to serve byte ranges to the audio element directly. The Service Worker workaround presented in Section \ref{sec:client-arch} works, but it has two consequences. First, Service Workers can only be registered on secure origins, so the web build needs HTTPS in production (\texttt{localhost} is exempt during development). Second, the cache on the web client is in-memory, so it is lost when the tab closes, which makes a web client a weaker peer seeder than a desktop client. A subtler point is that the Service Worker caps each response at 512\,000 bytes even if the browser asks for a larger range, because some Chromium versions issue an initial range request that asks for the entire song and would otherwise defeat the chunked progressive model.

\indent \par While WebRTC is a standard, the various platform implementations do not always negotiate cleanly with each other. The browser stack and the native libwebrtc stack that Flutter wraps differ in how they emit and consume ICE candidates: modern browsers obfuscate the local addresses behind mDNS hostnames (the \texttt{.local} candidates), which the native peers cannot always resolve, especially across operating systems with different mDNS support. The practical consequence is that an Android peer and a web peer sitting on the same Wi-Fi can sometimes fail to open a direct data channel and end up falling back to the server, even though they are physically a few metres apart. Desktop-to-desktop and desktop-to-mobile pairings tend to be more reliable than browser-to-mobile ones, but no exhaustive cross-matrix testing has been performed. The system simply relies on the server fallback whenever the negotiation fails, which keeps the playback experience identical for the user at the cost of a lower peer offload ratio in mixed-platform deployments.

\indent \par On mobile, acting as a peer seeder consumes the user's data allowance. The application has a setting to choose whether P2P runs only on Wi-Fi, only on cellular, or on both, and a per-month upload cap that is checked before sending a chunk. When the cap is reached, the chunk send requests are silently skipped and the requesting peer falls back to the server. This is a reasonable default, but it is not enforced from the server side: a user could in theory configure other peers to think this client has chunks that it then refuses to serve, which slightly degrades the global peer offload until the requesters time out. This is also mentioned in the future work section of Chapter \ref{chap:ch5}.

\indent \par The signaling WebSocket is not reconnected automatically. When the client loses the connection (network change, replica down, suspend/resume on a laptop), the current stream listener logs errors but does not clear the stored socket reference and start a new connection. In practice the device can remain invisible as a peer until the service is recreated, the user signs in again, or the application is restarted. A backoff-based reconnection loop, paired with re-registration of locally cached chunks after reconnect, is on the list of future improvements, as noted in Chapter \ref{chap:ch5}.

\indent \par The peer-assisted approach gives the largest cost reduction when many people listen to the same song around the same time. For long-tail content, where only a single client is listening at any given moment, the offload ratio drops to zero and the system behaves like a normal client-server stream. This is by design (the server fallback is always available and never blocks playback), but it does mean that the cost reduction predicted by the model in Chapter \ref{chap:ch2} is reached only for the popular fraction of the catalogue. Since real listening is heavily skewed toward a small set of popular tracks, that fraction is also where most of the egress, and therefore most of the potential saving, is concentrated.

\section{User Interface and Experience}
\label{sec:gui-manual}

\indent \par This last part returns from the network and service layers to the application as a listener sees it. This matters because the goal of the thesis is not only to reduce egress, but to do it without making playback feel experimental. The interface therefore stays close to the shape users already expect from a music player: a persistent navigation drawer on the left, the player bar at the bottom, and a central area that changes between catalogue, playback, upload and settings screens. The screenshots are taken from the macOS build with an admin user signed in, so they show both the ordinary listening flow and the administrative upload path that feeds the chunking pipeline described earlier.

\subsection{Home and Catalogue}
\label{subsec:gui-home-catalogue}

\indent \par Figure \ref{fig:screen-home} shows the home screen, with the drawer entries listed on the left (Home, Albums, Artists, Music, Playlists, Upload songs, Statistics, and User Settings at the bottom). The middle area is composed of three rails: 'Jump back in' with recently played albums, 'Recommended for you' with history-based suggestions, and 'Rediscover' with songs not listened to in a while. The player bar shows the current song, the lyrics line in the centre, and the standard playback controls on the right. The player bar is persistent across every screen, so playback never stops or loses its controls as the user moves through the catalogue.

\par The interface is intentionally ordinary, because the unusual part of the project is not a new way of pressing play, but what happens after that press. A listener should not have to know whether the next chunk came from the origin server, from the local cache or from another device on the swarm. The screens therefore keep the familiar shape of a music player, while the implementation underneath quietly switches between server bootstrap, peer delivery, cache hits and fallback. Hiding that machinery is itself a design goal, since the whole argument of the thesis is that egress can be moved off the origin without the listener paying for it in the experience.

\indent \par Now, the abstract roles from Chapter \ref{chap:ch2} have concrete code, services and screens behind them. The five roles, three interactions and three properties of the previous chapter can now be pointed at as running software rather than as a design on paper. The next question is no longer how the system is built, but whether those pieces actually reduce origin traffic when several listeners use the application at the same time. Answering it requires measurement rather than description, which is the task of the following chapter.
