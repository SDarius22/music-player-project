\indent \par Chapter 1 has identified the building blocks that a peer-assisted streaming system has to rest upon. The current chapter takes a step back and looks at each of these blocks more carefully, with the aim of understanding when they help and when they hurt. The discussion here is intentionally less applied: there is no Spring Boot or Flutter yet, only the equations and the trade-offs that govern cost, latency, and integrity of any system that fits in this category. What I want from this chapter is that, by the time the reader gets to the end, the implementation choices in Chapter \ref{chap:ch3} require no further justification.

\par I will also use the last section of the chapter to describe an abstraction of the system, before all the technologies and the framework names are layered on top of it. This is useful because it isolates the conceptual roles from their concrete realisation, which makes it easier to reason about correctness and about replaceability later on.


\section{Formalizing the Cost Problem}
\label{sec:delivery-models}

\indent \par The original motivation of this paper, presented in Chapter 1, is the rapidly growing cost of cloud egress. The discussion was qualitative there. In what follows, I model it more rigorously, so that the gain promised by the peer-assisted approach can be measured and not only claimed.

\subsection{The Pure Client-Server Baseline}
\label{subsec:client-server}

\indent \par In the simplest delivery architecture, every client opens a direct TCP connection to the origin server and pulls data at the required bitrate. The model is easy to reason about. The server holds all content, all the bandwidth is paid by the provider, and the clients are purely passive consumers. The fundamental issue is that the server egress cost grows linearly with the number of concurrent listeners. If a song of size $S$ bytes is listened to by $N$ concurrent users, the total egress volume is

\begin{equation}
  B_{\text{server}} = N \cdot S.
  \label{eq:server-cost}
\end{equation}

\par For large $N$, this is commercially unsustainable, exactly the situation that Spotify is currently in \cite{spotify_google}. Cloud egress pricing from major providers (AWS, GCP, Azure) is typically charged per gigabyte transferred out of the data centre, so equation \eqref{eq:server-cost} translates directly into an infrastructure bill that scales with popularity.

\subsection{Content Delivery Networks}
\label{subsec:cdn}

\indent \par Content Delivery Networks (CDNs) try to address the latency dimension of the client-server problem by replicating popular content to geographically distributed edge nodes. A client is routed to the closest edge node, which reduces the round-trip time and often improves the throughput. CDNs do not reduce the total egress though. Every byte still originates from infrastructure that the provider owns or rents and is billed at egress rates, with the replication itself adding intra-CDN transfer fees for warming the edge caches. The model is also poorly suited for long-tail content (songs that are rarely played) which has to be replicated to many edge locations even though it is requested only occasionally. CDNs therefore solve in part the latency problem while leaving the cost problem intact \cite{web_performance}.

\subsection{Peer-Assisted Hybrid Delivery}
\label{subsec:hybrid}

\indent \par The peer-assisted model adds a device-to-device transfer layer on top of the client-server baseline \cite{d2d_survey}. After a client downloads a portion of the content from the server, it advertises that portion to other interested clients and serves it directly over a P2P channel. The server shifts from serving every byte to serving only the initial fraction that bootstraps each new client into the swarm. The theoretical minimum server egress for a swarm of $N$ users is reached when the first client downloads the full content from the server and every subsequent client downloads only from peers:

\begin{equation}
  B_{\text{hybrid}}^{\min} = S + (N-1) \cdot \epsilon,
  \label{eq:hybrid-min}
\end{equation}

where $\epsilon$ is the per-client server overhead used for bootstrapping (the prefix). In practice, churn (clients that leave before peers manage to download from them) raises this floor, but the gain over equation \eqref{eq:server-cost} remains substantial even under realistic churn rates \cite{measurement_p2p}. Figure \ref{fig:cost-comparison} illustrates the asymptotic difference between the two models.

\par The hybrid model brings design constraints that the pure server model does not have. Peers must be discoverable, must hold the relevant data at the right moment, and must be reachable despite NAT and firewall boundaries. The rest of this chapter addresses each of these constraints in turn.


\section{Chunks, Hashes and Deduplication}
\label{sec:chunk-theory}

\indent \par Chapter 1 already argued that the chunk-based approach is the only sensible choice for a P2P system. This section makes the argument quantitative, by showing how the chunk size, the cryptographic hash and the integrity check fit together.

\subsection{Choosing the Chunk Size}
\label{subsec:fixed-chunks}

\indent \par Fixed-size partitioning of an audio file in blocks of $C$ bytes produces $\lceil S / C \rceil$ chunks. The choice of $C$ is a tuning parameter that trades latency for overhead. Smaller chunks reduce the time-to-first-byte of a P2P transfer but increase the number of round trips and the size of the index. Larger chunks amortise the per-request overhead but coarsen the granularity at which a peer can serve partial data.

\par A size of $C = 65\,536$ bytes (64 KiB) sits at the intersection of efficient I/O alignment (a power-of-two multiple of typical disk and network buffer sizes) and practical P2P scheduling granularity. At a typical audio bitrate of 320 kbps for a high-quality MP3, one chunk holds approximately 1.6 seconds of audio, a resolution that is finer than the human perception of buffering artefacts (around two to three seconds) while still being coarse enough to keep the chunk count per song in the tens to low hundreds. The same value also has a precedent in BitTorrent's default piece sizes, which were tuned for the same balance \cite{bittorent}.

\subsection{Content-Addressed Storage}
\label{subsec:cas}

\indent \par Content-addressed storage (CAS) is a paradigm in which the identity of a data block is derived from its content rather than from an externally assigned identifier or file path. The canonical mechanism is to use a cryptographic hash of the block's bytes as the key. This thesis uses SHA-256, which maps any input to a 256-bit digest and has a collision probability of approximately $2^{-256}$, negligible in any practical context.

\par The deduplication benefit of CAS is directly applicable to a music library. Many tracks share silence segments, common fade-outs, or identical metadata headers embedded in the audio container. With CAS, all such identical 64 KiB blocks are stored only once, regardless of how many songs reference them. Formally, if $k$ distinct byte-level 64 KiB blocks occur across all songs, the storage footprint is $k \cdot C$ bytes, which is at most $\sum_i S_i$ (the sum of all song sizes) and often substantially less. The duplicates that hide inside the long tail of any user's library are surprisingly numerous, especially when the same song was downloaded from two different sources.

\subsection{Hash-Based Integrity Verification}
\label{subsec:integrity}

\indent \par The same SHA-256 digest that deduplicates the storage is also used as an integrity check. Because the digest is precomputed at upload time and stored in the chunk manifest, a client receiving chunk $i$ from any source (origin server or peer) can recompute the SHA-256 of the received bytes and compare it with the expected value. A mismatch unambiguously indicates corruption or deliberate tampering. This property is critical in the peer-assisted model, where a misbehaving or compromised peer could otherwise inject corrupted audio data \cite{webrtc_security}. The verification cost is negligible relative to the network latency, which means that we get the integrity check for free in the cost model.


\section{WebRTC: From Theory to Numbers}
\label{sec:webrtc-theory}

\indent \par Web Real-Time Communication, already introduced in Chapter 1, is the only viable transport for cross-platform peer connectivity. Its adoption in this system is motivated by three properties: ubiquity (it is supported natively by every major browser, and Flutter exposes it through the \texttt{flutter\_webrtc} package), the ability to traverse NATs, and the support for ordered reliable data channels \cite{latency_and_reliability}.

\subsection{NAT Traversal: ICE, STUN, TURN}
\label{subsec:nat-traversal}

\indent \par The majority of end-user devices sit behind Network Address Translation (NAT) gateways, which rewrite the source IP address and the port of the outgoing packets. A device behind NAT cannot be contacted directly by an external peer using its private address, so a traversal mechanism is needed.

\par Interactive Connectivity Establishment (ICE) is the WebRTC framework for discovering and negotiating a viable network path between two peers. Given that it has the word ``establishment'' in its name, it should be no surprise that it takes its time. ICE collects candidates (possible transport addresses) from three sources:

\begin{enumerate}
  \item \textbf{Host candidates}: the device's own network interface addresses.
  \item \textbf{Server-reflexive candidates}: the public IP/port mapping created by the NAT, discovered by querying a STUN (Session Traversal Utilities for NAT) server.
  \item \textbf{Relayed candidates}: addresses on a TURN (Traversal Using Relays around NAT) relay server, used only when direct connectivity fails \cite{relays_turn}.
\end{enumerate}

\par In most residential NAT configurations, a STUN exchange is enough. Both peers learn their server-reflexive addresses and use UDP hole-punching to establish a direct path \cite{top_down_cn}. TURN relaying is reserved for symmetric NATs and enterprise firewalls. Its cost is significantly higher, but the traffic volume is bounded because the system falls back to server delivery rather than to TURN whenever P2P fails. \subsection{Data Channels and Binary Transfer}
\label{subsec:data-channels}

\indent \par WebRTC Data Channels provide an SCTP-over-DTLS transport that supports both reliable ordered delivery (similar to TCP) and unreliable unordered delivery (similar to UDP). For audio chunk transfer, the reliable ordered mode is the appropriate choice. The audio data has to arrive uncorrupted and the chunk boundary semantics depend on receiving complete 64 KiB blocks. Unlike TCP, however, the SCTP head-of-line blocking is confined to a single stream, so a stalled chunk request on one data channel does not block data channels to other peers \cite{webrtc_testing}.

\par The binary message format over a data channel is intentionally minimal. A short fixed header encodes the file hash and the chunk index, followed directly by the audio payload. Choosing a fixed-length header over a framing protocol such as Protocol Buffers is motivated by the fact that the chunk transfers are entirely homogeneous. There is no multiplexing of different message types on a single data channel, so a length-prefixed header would only add overhead without bringing any benefit.

\subsection{Why a Server is Still Needed}
\label{subsec:signaling-role}

\indent \par WebRTC requires an out-of-band signaling channel for the exchange of SDP (Session Description Protocol) offer/answer payloads and ICE candidates. This channel is the only persistent server dependency on the P2P path. Once the data channel is open, the server is no longer involved in the media transfer. Choosing WebSocket as the signaling transport aligns well with the existing server infrastructure and lets the same connection also carry library synchronisation events and peer discovery messages, which reduces the number of persistent connections each client has to maintain.


\section{Prefix Caching as a Bootstrap Strategy}
\label{sec:prefix-theory}

\indent \par Without a mitigation, a peer-assisted streaming system has a chicken-and-egg problem: a new client cannot receive data from peers until it has connected to them, and connecting to peers takes the few hundred milliseconds described above. During this window, the audio player has nothing to decode and the playback cannot start. This is exactly the situation that earlier proxy prefix caching schemes \cite{prefix_caching} were created to mitigate in the unicast world.

\subsection{Decomposing the Startup Latency}
\label{subsec:latency-decomp}

\indent \par The total startup latency $T_{\text{start}}$ for a client requesting a song can be decomposed as

\begin{equation}
  T_{\text{start}} = T_{\text{resolve}} + T_{\text{buffer}},
  \label{eq:latency}
\end{equation}

where $T_{\text{resolve}}$ is the time needed to acquire enough data for the audio decoder to begin, and $T_{\text{buffer}}$ is the time needed to buffer a sufficient lookahead so the decoder does not starve afterwards. In a peer-only model, $T_{\text{resolve}}$ is dominated by the WebRTC connection setup time ($T_{\text{ICE}} \approx 200$ to $500$ ms) plus the first chunk transfer latency. An unmitigated peer-first strategy would therefore introduce a mandatory 200 to 500 ms delay even for popular songs with many available peers, which is well known to drive listeners away \cite{quality_impact}.

\par The prefix strategy eliminates this delay by decoupling the initial buffer fill from the peer discovery. The client simultaneously fetches a fixed prefix of the song from the origin server over HTTP, and initiates peer discovery via the signaling server. The playback begins as soon as the prefix arrives, and by the time the prefix is consumed, the peer connections are typically established and the rest of the song is served from peers.

\subsection{Choosing the Prefix Threshold}
\label{subsec:prefix-threshold}

\indent \par The prefix has to be large enough that $T_{\text{buffer}}$ covers the full peer negotiation time $T_{\text{ICE}}$ plus the first peer response time. Let $r$ be the audio bitrate in bytes per second and $P$ the prefix size in bytes. The prefix sustains the playback for $P / r$ seconds. For this to cover the ICE negotiation and an initial P2P timeout of $T_{\text{p2p}}$ seconds, we need

\begin{equation}
  \frac{P}{r} > T_{\text{ICE}} + T_{\text{p2p}}.
  \label{eq:prefix-condition}
\end{equation}

\par At $r = 40\,960$ bytes/s (320 kbps), $T_{\text{ICE}} = 500$ ms, and $T_{\text{p2p}} = 1\,000$ ms, equation \eqref{eq:prefix-condition} gives $P > 61\,440$ bytes, which is less than one chunk. The implementation goes beyond this minimum and uses a dynamic prefix that scales with the song length, providing a generous safety margin in slow NAT environments. The exact rule will be presented in Chapter \ref{chap:ch3}.

\subsection{Predictive Pre-Fetching as a Companion}
\label{subsec:proactive-theory}

\indent \par A user that listens to a playlist will transition between songs with negligible delay only if the next song's startup sequence (prefix fetch and peer negotiation) completes before the current song ends. Predictive Pre-Fetching, which Chapter 1 introduced, fits naturally as a companion to prefix caching: it initiates prefix downloads for the next $k$ songs in the queue while the current song is still playing. The bandwidth overhead is bounded. The total proactive download is $k \cdot P$ bytes, which for a small $k$ and a sub-megabyte prefix amounts to a modest background cost compared to the elimination of any inter-song latency \cite{limits_prefetching}.


\section{A Bandwidth Cost Model}
\label{sec:cost-model}

\indent \par Having established the individual mechanisms, this section combines them in a formal cost model that quantifies the expected reduction in server egress and motivates the parameter choices made in the implementation.

\subsection{Theoretical Offload Ratio}
\label{subsec:offload-ratio}

\indent \par I define the offload ratio $\rho$ as the fraction of total bytes served by peers rather than by the origin server. For a swarm of $N$ simultaneous listeners of a song of size $S$, every listener needs a prefix of size $P$ from the server. Assuming all non-prefix bytes are served by peers (the ideal case), the server egress is $N \cdot P$ and the total bytes consumed are $N \cdot S$, so

\begin{equation}
  \rho = 1 - \frac{P}{S}.
  \label{eq:offload-ratio}
\end{equation}

\par For a 4-minute song at 320 kbps, $S \approx 9.6$ MB. With a prefix of about half a megabyte, $\rho \approx 94.6\%$. Even a 1-minute track with $S \approx 2.4$ MB yields $\rho \approx 78.3\%$. These figures are upper bounds. In practice, the churn and the cold starts reduce the achievable ratio, but the asymptotic behaviour in equation \eqref{eq:server-cost} is converted from $O(N)$ to $O(1)$ in server egress per unit of time. The cost becomes effectively independent of the swarm size.

\subsection{The Trade-offs the Model Exposes}
\label{subsec:tradeoffs}

\indent \par The model reveals several trade-offs that informed the implementation:

\begin{itemize}
  \item \textbf{Prefix size vs. offload ratio}: A larger prefix reduces $\rho$ (more server cost), but it increases the startup robustness in slow networks.

  \item \textbf{P2P timeout vs. stall probability}: A longer timeout gives slow peers more time to respond, which increases the offload ratio. In the same time, it risks audible stalls if the audio player exhausts its lookahead buffer. A practical timeout combined with a multi-chunk lookahead ensures the buffer survives at least one failed peer request before the server fallback returns the data.

  \item \textbf{Chunk size vs. overhead}: Halving $C$ to 32 KiB doubles the chunk count and the index size while reducing the per-chunk transfer latency. Doubling $C$ to 128 KiB reduces the overhead but coarsens the P2P scheduling granularity, making it harder for a peer to serve partial data when it has only a part of a large block. The 64 KiB value is a practical optimum for the target content (320 kbps MP3 and FLAC).

  \item \textbf{Swarm size and peer density}: The model assumes that at least one peer with a given chunk is reachable within the P2P timeout. For long-tail songs with very few listeners, peers may simply be absent, and the system gracefully degrades to pure server delivery. No additional logic or user experience penalty is incurred. The server fallback is always available.
\end{itemize}

\section{An Abstract View of the System}
\label{sec:system-abstraction}

\indent \par Before going to the concrete details of Chapter \ref{chap:ch3}, it is useful to describe the system at a level that hides the choice of framework, language, and database. This abstraction also makes it easier to argue why each role exists, independently of the technology used to realise it.

\subsection{The Five Logical Roles}
\label{subsec:abstract-roles}

\indent \par The system is composed from five logical roles. They are independent of the technology used to realise them.

\begin{itemize}
  \item \textbf{Catalogue and identity store}: a transactional store that keeps the metadata of songs, albums, artists and playlists, the registration of users, and the per-user library state. It is the only durable source of truth for who a user is, what they have, and which content exists in the system.

  \item \textbf{Chunk store}: a content-addressed store that keeps the binary audio data, indexed by the SHA-256 of its content. Every song is mapped to an ordered list of chunk references. Two songs that share an identical 64 KiB block reference the same physical chunk.

  \item \textbf{Origin server}: a stateless service that sits in front of the two stores and exposes a manifest endpoint (which gives the ordered list of chunk hashes for a song), the individual chunk endpoints, and the catalogue REST API.

  \item \textbf{Coordination bus}: an in-memory pub/sub system, plus a small key-value index, that holds the ephemeral state of the currently connected peers. This is what allows a peer on one origin server replica to learn about a peer on another replica, without the two replicas having to know each other directly.

  \item \textbf{Client}: the application running on the user's device. The client owns the audio player, the local chunk cache, the WebRTC peer connections, and a small amount of UI state. It is the only component that produces audio output.
\end{itemize}

\subsection{The Three Abstract Interactions}
\label{subsec:abstract-interactions}

\indent \par The five roles are tied together by three abstract interactions.

\par \textbf{Discovery.} When the client decides to play a song, it asks the origin server for the manifest of that song. In parallel, it asks the coordination bus which other peers have already cached chunks of the same song. The bus returns a peer-to-chunk map, and the client uses this map to choose where to look first.

\par \textbf{Transfer.} Each chunk is fetched from one of three sources, in order of preference. The local cache is consulted first (zero network cost). If the chunk is not cached and its index falls inside a small prefix range, the origin server is used (predictable latency, bootstraps the playback). Otherwise the client tries the best-ranked peer first. If the peer fails to respond within a short timeout, the origin server is used as a fallback. Every chunk that arrives is verified against the hash from the manifest before being given to the audio decoder.

\par \textbf{Coordination.} Whenever a chunk has been cached locally, the client tells the coordination bus, so future peers can find it. Whenever a server-side change happens that another device of the same user should know about (a library edit, a playback state change), the origin server publishes a small notification on the bus. The other devices, which are subscribed through their signaling connection, receive the notification and react.

\subsection{Properties to Preserve}
\label{subsec:abstract-properties}

\indent \par The abstraction has three properties that the implementation in the next chapter has to preserve, regardless of the chosen technologies:

\begin{itemize}
  \item \textbf{Replica symmetry}: every replica of the origin server is equivalent. A client may connect to any of them, and the coordination bus is what makes the cross-replica routing transparent.

  \item \textbf{Graceful degradation}: every part of the system has a well-defined fallback. If no peers are available, the server delivers everything. If the local cache is cold, the prefix is fetched from the server. If a peer responds with a corrupted chunk, the server is asked again. The user sees no error in any of these cases.

  \item \textbf{No trusted peer}: peers are never authoritative. A peer chunk is valid only if its SHA-256 matches the manifest hash that was fetched from the origin server. A malicious peer cannot inject bad audio data without being detected.
\end{itemize}

\par With this abstraction in place, Chapter \ref{chap:ch3} can be read as a long answer to a single question: how do these five roles, three interactions and three properties translate into actual code?
