ABSTRACT
\vspace{0.5cm}	
\hrule
\vspace{0.5cm}	
%\cleardoublepage
On-demand music streaming has become economically difficult to sustain. Cloud egress is billed per gigabyte and scales linearly with the number of concurrent listeners, and the rapid growth of AI workloads has pushed cloud prices even higher. This thesis designs, implements and evaluates a peer-assisted music streaming system that converts the per-listener server cost from $O(N)$ to a value bounded by $O(1)$ in the ideal case, without sacrificing the near-instant playback that users expect.

\par The first chapter surveys the five design domains that a peer-assisted streaming system has to address (latency, connectivity, storage, integrity and flow control) and picks, for each one, the candidate solution best suited to a music context: prefix caching, WebRTC, mobile edge caching, chunk-based synchronization and adaptive routing. The second chapter justifies those choices quantitatively, deriving a bandwidth cost model, the 64 KiB chunk size and the 5\% prefix threshold, and ending with an abstract view of the system that separates the logical roles from the technologies that realise them.

\par The third chapter is the implementation. A cross-platform music player is built on Flutter and Java Spring Boot, with PostgreSQL and Redis as the persistence and coordination layer. Audio files are split into 64 KiB chunks identified by their SHA-256 hash, deduplicated on the server and verified on the client regardless of where they come from. A server-side prefix of about 5\% of the song masks the WebRTC connection setup time, while the remaining bytes flow directly between peers over reliable data channels, with the server always available as a fallback. The same code base runs unchanged on Linux, macOS, Windows, Android, iOS and the web browser.

\par The original contribution lies in combining these five trade-offs into a single working system that uses no TURN relay (preserving the cost-saving property of the peer model), runs across the six target platforms from one source tree, and is small enough to be operated on commodity hardware while still demonstrating the egress reduction predicted by the cost model.

INTRODUCTION
\vspace{0.5cm}
\hrule
\vspace{0.5cm}

Every time a song plays on a cloud-hosted streaming service, the bytes that reach the listener are paid for twice over: once to store and process them, and again, per gigabyte, as they leave the data centre. That second cost, the egress, scales with how many people are listening rather than with the revenue they bring, so a track that suddenly becomes popular becomes, at the same moment, more expensive to serve. The large providers absorb this with scale and long-term cloud commitments, but the underlying curve is unforgiving: double the audience and you double the bill.

\par There is, however, a delivery network already sitting idle inside the problem. Every listener who has just received a piece of a song holds a perfectly good copy of it, and the listener starting the same song a moment later needs exactly that. If clients could hand those pieces to one another, the origin server would shrink from the source of every byte into a bootstrapper that seeds the swarm and then stands by as a fallback. The difficulty is doing this without the listener ever noticing: playback has to start instantly, the same code has to run in a browser and on a phone, and a piece of audio received from an untrusted peer has to be as safe as one received from the server. This thesis designs, builds and measures a music streaming system that meets those constraints, turning the per-listener server cost from $O(N)$ into a quantity bounded by $O(1)$ in the ideal case.

\par The thesis is organised into four chapters. Chapter \ref{intro} sets out the problem context and surveys the related work, breaking peer-assisted streaming into five design domains (latency, connectivity, storage, integrity and flow control) and weighing the candidate solutions for each. Chapter \ref{chap:ch2} turns those choices into a formal cost model and derives the system's key parameters, among them the 64 KiB chunk and the dynamic 5\% prefix. Chapter \ref{chap:ch3} is the implementation, presented in three parts: the architecture and its layers, the implementation of the runtime flows together with how they were tested, and the user interface. Chapter \ref{chap:ch4} evaluates the working prototype, measuring the egress offload, the startup latency and how the system scales as the audience grows, and placing the results next to the systems from the literature.

\par The original contributions of this work are the following:
\begin{itemize}
  \item a single Flutter and Spring Boot code base that runs the same peer-assisted streaming client across six platforms (Linux, macOS, Windows, Android, iOS and the web browser, the last through a Service Worker);
  \item a dynamic prefix-caching bootstrap, serving roughly 5\% of each song from the origin, that hides the WebRTC connection setup so that playback starts immediately while the peer mesh forms in the background;
  \item a 64 KiB content-addressed (SHA-256) chunking scheme that deduplicates storage on the server and lets the client verify every chunk identically, whether it arrived from a peer or from the origin;
  \item a WebRTC data-channel peer transport with round-trip-time- and throughput-based peer ranking, staggered fan-out and an always-present server fallback, deliberately built without a TURN relay so that the cost-saving property of the peer model is preserved;
  \item prefix-only predictive prefetching of upcoming queue entries, which removes the gap between songs at a bounded bandwidth cost;
  \item a multi-replica backend whose cross-replica peer signaling is made transparent to the client through Redis publish/subscribe;
  \item an end-to-end evaluation, built on a purpose-made headless measurement client, that quantifies the egress saving, validates the cost model, and in the process uncovered and fixed a signaling concurrency bottleneck that had been capping multi-peer scaling.
\end{itemize}
