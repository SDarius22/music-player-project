ABSTRACT
\vspace{0.5cm}	
\hrule
\vspace{0.5cm}	
%\cleardoublepage
On-demand music streaming has become economically difficult to sustain. Cloud egress is billed per gigabyte and scales linearly with the number of concurrent listeners, and the rapid growth of AI workloads has pushed cloud prices even higher. This thesis designs, implements and evaluates a peer-assisted music streaming system that converts the per-listener server cost from $O(N)$ to a value bounded by $O(1)$ in the ideal case, without sacrificing the near-instant playback that users expect.

\par The first two chapters survey the design space of peer-assisted streaming and turn the chosen mechanisms into a cost model that fixes the 64 KiB chunk size and the dynamic 5\% prefix. The third chapter implements the system as a single Flutter and Spring Boot code base, backed by PostgreSQL and Redis, that runs unchanged on Linux, macOS, Windows, Android, iOS and the web browser: audio is split into SHA-256-addressed chunks that flow directly between peers over WebRTC, the server serves only a small prefix and stays available as a fallback, and no TURN relay is used, so the cost-saving property of the peer model is preserved.

\par The fourth chapter evaluates the prototype. A warm swarm of two to five listeners served 88.4\% of its bytes from peers, the origin share stayed flat as the audience grew from one listener to five (the concrete form of the $O(1)$ claim), and the 5\% prefix hid the median 243 ms connection setup, so click-to-sound stayed at the level of an ordinary server stream; these are best-case figures from a single-machine, cold-cache setup rather than a production measurement. The fifth chapter concludes that the listeners of a song are collectively most of the infrastructure needed to deliver it, that this can be exploited while keeping playback instant, the catalogue safe and the code single-sourced, and it lays out the wide-area, warm-cache and mixed-platform experiments left for future work.

INTRODUCTION
\vspace{0.5cm}
\hrule
\vspace{0.5cm}

Every time a song plays on a cloud-hosted streaming service, the bytes that reach the listener are paid for twice over: once to store and process them, and again, per gigabyte, as they leave the data center. That second cost, the egress, scales with how many people are listening rather than with the revenue they bring, so a track that suddenly becomes popular becomes, at the same moment, more expensive to serve. The large providers absorb this with scale and long-term cloud commitments, but the underlying cost is unforgiving: double the audience and you double the bill.

\par There is, however, a delivery network already sitting idle inside the problem. Every listener who has just received a piece of a song holds a perfectly good copy of it, and the listener starting the same song a moment later needs exactly that. If clients could hand those pieces to one another, the origin server would shrink from the source of every byte into a helper that seeds the swarm and then stands by as a fallback. The difficulty is doing this without the listener ever noticing: playback has to start instantly, the same code has to run in a browser and on a phone, and a piece of audio received from an untrusted peer has to be as safe as one received from the server. This thesis designs, builds and measures a music streaming system that meets those constraints, turning the per-listener server cost from $O(N)$ into a quantity bounded by $O(1)$ in the ideal case.

\par The thesis is organized into five chapters. Chapter \ref{intro} sets out the problem context and surveys the related work, breaking peer-assisted streaming into five design domains (latency, connectivity, storage, integrity and flow control) and weighing the candidate solutions for each. Chapter \ref{chap:ch2} turns those choices into a formal cost model and derives the system's key parameters, among them the 64 KiB chunk and the dynamic 5\% prefix. Chapter \ref{chap:ch3} is the implementation, presented in three parts: the architecture and its layers, the implementation of the runtime flows together with how they were tested, and the user interface. Chapter \ref{chap:ch4} evaluates the working prototype, measuring the egress offload, the startup latency and how the system scales as the audience grows, and placing the results next to the systems from the literature. Chapter \ref{chap:ch5} draws the conclusions and sets out the directions for future work.

\par The original contributions of this work are the following:
\begin{itemize}
  \item a single Flutter and Spring Boot code base that runs the same peer-assisted streaming client across six platforms;
  \item a dynamic prefix-caching bootstrap, serving roughly 5\% of each song from the origin, that hides the WebRTC connection setup so that playback starts immediately while the peer mesh forms in the background;
  \item a 64 KiB content-addressed (SHA-256) chunking scheme that deduplicates storage on the server and lets the client verify every chunk identically, whether it arrived from a peer or from the origin;
  \item a WebRTC data-channel peer transport with round-trip time and throughput-based peer ranking, staggered fan-out and an always-present server fallback, deliberately built without a TURN relay so that the cost-saving property of the peer model is preserved;
  \item prefix-only predictive prefetching of upcoming queue entries, which removes the gap between songs at a bounded bandwidth cost;
  \item a multi-replica backend whose cross-replica peer signaling is made transparent to the client through Redis publish/subscribe;
  \item an end-to-end evaluation, built on a purpose-made headless measurement client, that quantifies the egress saving, validates the cost model, and in the process uncovered and fixed a signaling concurrency bottleneck that had been capping multi-peer scaling.
\end{itemize}
